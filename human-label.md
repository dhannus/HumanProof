# Human Label Specification — v0.1 (Draft)

> Status: **Draft — open for community review**  
> This spec is intentionally incomplete. Open an issue to propose changes.

---

## Purpose

The Human Label is a short-lived, cryptographic token that proves a real human initiated an online action — without revealing who that human is.

It is the core primitive of the HumanProof protocol.

---

## Design Goals

| Goal | Description |
|------|-------------|
| **Privacy** | No personal data in the label. No linkability across sessions. |
| **Verifiability** | Any platform can verify the label is valid without contacting a central authority. |
| **Expiry** | Labels are short-lived. A new label is generated each session. |
| **Extensibility** | Optional fields (age group, scope) can be added without breaking v0.1 parsers. |
| **Auditability** | Labels can be resolved to a real identity ONLY via court order + independent trustee. |

---

## Label Format

### JSON Representation

```json
{
  "version":    "0.1",
  "type":       "human-label",
  "proof":      "<zkp_proof_string>",
  "issued_at":  1743000000,
  "ttl":        3600,
  "expires_at": 1743003600,
  "scope":      "session",
  "age_group":  null,
  "poc":        false
}
```

### Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | string | ✅ | Protocol version. Always `"0.1"` for this spec. |
| `type` | string | ✅ | Always `"human-label"`. Distinguishes from Machine Token. |
| `proof` | string | ✅ | ZKP proof string. In PoC: simulated hex string. In production: Groth16 or PLONK proof (~200–500 bytes base64). |
| `issued_at` | integer | ✅ | Unix timestamp (seconds) when the label was generated. |
| `ttl` | integer | ✅ | Time-to-live in seconds. Default: `3600` (1 hour). |
| `expires_at` | integer | ✅ | `issued_at + ttl`. Convenience field for fast expiry check. |
| `scope` | string | ✅ | Always `"session"` in v0.1. Future: `"action"` for single-action labels. |
| `age_group` | string\|null | — | Optional. `"under_18"` \| `"18_plus"` \| `null`. Only disclosed when user consents and platform requires it. |
| `poc` | boolean | — | If `true`, this label was generated by a PoC client and its proof is simulated. Platforms SHOULD treat `poc: true` labels as unverified in production. |

## Wire Format

Labels are transmitted as **Base64-encoded JSON** in an HTTP request header:

```
X-HumanProof-Label: eyJ2ZXJzaW9uIjoiMC4xIiwidHlwZSI6Imh1bWFuLWxhYmVsIi...
X-HumanProof-Version: 0.1
```

## Verification

### Platform Verification Flow

```
1. Decode Base64 → parse JSON
2. Check type == "human-label"
3. Check version is supported
4. Check expires_at > current timestamp
5. If poc == true → reject or flag (in production environments)
6. Verify ZKP proof against public parameters
7. Return: VALID | EXPIRED | INVALID | POC_ONLY
```

### Minimal Verification (PoC)

In the PoC phase, platforms can perform steps 1–5 only (no real ZKP verification):

```javascript
function verifyLabel(base64Label) {
  try {
    const label = JSON.parse(atob(base64Label));
    if (label.type !== 'human-label') return 'INVALID';
    if (Date.now() / 1000 > label.expires_at) return 'EXPIRED';
    if (label.poc) return 'POC_ONLY';
    // TODO: real ZKP verification here
    return 'VALID';
  } catch {
    return 'INVALID';
  }
}
```

## Privacy Properties

### Unlinkability
Labels generated in different sessions are computationally unlinkable.  
The ZKP proof is re-randomised each time, even from the same credential.

### No User Identifier
The label contains no user ID, no email, no IP address, no device fingerprint.

### Minimal Disclosure
The `age_group` field is `null` by default and only populated when:
1. The user explicitly consents, AND
2. The platform has a legitimate reason (age-restricted content)

### Trustee Resolution
A label can ONLY be traced back to a real identity by:
1. A court order with specific initial suspicion
2. A request to the independent trustee (not the platform, not the government directly)
3. The trustee resolving the encrypted credential mapping

This process is logged in the public transparency report.

## Label Lifecycle

```
User opens browser
       │
       ▼
Extension prompts verification
       │
       ▼
ZKP proof generated from local credential
(credential never leaves device)
       │
       ▼
Human Label issued (TTL: 1 hour)
       │
       ├──→ Label attached to outgoing requests (X-HumanProof-Label header)
       │
       ├──→ Platforms verify label on receipt
       │
       ▼
TTL expires → Label deleted from storage → new label required next session
```

## Known Limitations (v0.1)

- Labels are per-device. Multi-device use requires separate verification on each device.
- A human can share their verified device. The protocol does not prevent this.
- The PoC uses a simulated proof — not cryptographically secure.
- No cross-platform label standard yet (eIDAS 2.0 alignment planned for v0.3).
